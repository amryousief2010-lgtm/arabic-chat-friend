## نظام المجزر ومصنع اللحوم — Capital Ostrich

سأبني نظامين متكاملين مرتبطين تلقائياً بالمزرعة والمخزون والمبيعات، باستخدام الباركودات الرسمية (COC) المعتمدة لديكم.

---

### القرارات الافتراضية (بناءً على واقع شركتكم)

- **مصدر الطيور**: مزدوج (المزرعة الداخلية + موردون خارجيون).
- **المخرجات**: جدول قياسي (Yield Standards) لكل قطعة + إدخال فعلي لكل دفعة + تقرير الفروقات (Variance Analysis).
- **منتجات المصنع**: لحم مفروم، برجر نعام، نقانق، شاورما متبلة، كفتة (قابلة للتوسع).
- **الربط**: تلقائي كامل (مجزر → مخزون لحوم → مصنع → مخزون منتجات → مبيعات).

---

### المرحلة 1: قاعدة بيانات المجزر (Slaughterhouse)

```text
slaughter_live_receipts        استلام طيور حية (مصدر، عدد، وزن، سعر/كجم)
slaughter_batches              دفعات الذبح (تاريخ، شيفت، عدد، نافق، مرفوض)
slaughter_yield_standards      جدول المخرجات القياسية لكل قطعة (% من الوزن الحي)
slaughter_batch_outputs        المخرجات الفعلية لكل دفعة (قطعة + وزن فعلي + باركود)
slaughter_workers              عمال المجزر + إنتاجية يومية
slaughter_quality_checks       فحوصات بيطرية/جودة قبل وبعد الذبح
```

كل دفعة ذبح تنشئ تلقائياً:
- خصم الطيور الحية من رصيد المصدر
- إضافة المخرجات إلى مخزون اللحوم الطازجة بالباركود الرسمي
- حساب التصافي الفعلي مقابل المعياري
- حساب تكلفة الكيلو لكل قطعة (تكلفة الطائر / إجمالي المخرجات بوزن الكيلو)

### المرحلة 2: قاعدة بيانات مصنع اللحوم (Meat Factory)

```text
meat_factory_products          المنتجات المصنعة (اسم، باركود، عبوة، صلاحية)
meat_factory_recipes (BOM)     وصفة كل منتج (لحم خام + توابل + إضافات)
meat_factory_batches           أوامر إنتاج (دفعات تصنيع)
meat_factory_consumption       استهلاك المواد لكل دفعة
meat_factory_quality_checks    فحوصات الجودة على المنتج النهائي
meat_factory_packaging         التعبئة (عدد العبوات + ملصقات الباركود)
```

كل دفعة تصنيع:
- تخصم اللحوم الخام من مخزون المجزر تلقائياً
- تخصم التوابل والإضافات من المخزون
- تنتج كميات المنتج النهائي بتاريخ إنتاج وصلاحية
- تحسب التكلفة الفعلية (مواد + عمالة + overhead)

### المرحلة 3: الصلاحيات (RBAC)

نستخدم الأدوار الموجودة:
- `slaughterhouse_manager` → إدارة كاملة للمجزر
- `meat_factory_manager` → إدارة كاملة للمصنع
- `quality_manager` → إضافة فحوصات جودة على الاثنين
- `warehouse_supervisor` → استلام المخرجات
- `general_manager` / `executive_manager` → كل شيء
- باقي الأدوار: قراءة فقط

### المرحلة 4: واجهات المستخدم

**صفحة المجزر `/modules/slaughterhouse`:**
- لوحة KPI (طيور اليوم، تصافي اليوم، إنتاج الشهر، تكلفة الكيلو)
- تبويبات: استلام حي / دفعات ذبح / المخرجات / الجودة / التقارير
- نموذج دفعة ذبح بشاشة سريعة (إدخال أوزان كل قطعة دفعة واحدة)
- تقرير الفروقات (معياري vs فعلي) برسوم بيانية

**صفحة مصنع اللحوم `/modules/meat-factory`:**
- لوحة KPI (إنتاج اليوم، تكلفة المنتج، هامش الربح)
- تبويبات: المنتجات / الوصفات / أوامر الإنتاج / الجودة / التعبئة
- محرر وصفات (BOM) بـ Drag & Drop
- شاشة بدء دفعة تصنيع → تخصم المواد تلقائياً وتطبع ملصقات

**صفحة مشتركة `/modules/meat-flow`:**
- خريطة تدفق اللحوم من المزرعة → المجزر → المصنع → المخزون → البيع
- تتبع الـ batch من البداية للنهاية (Traceability)

### المرحلة 5: الذكاء الاصطناعي

- **تحليل التصافي**: عند انخفاض التصافي، Gemini يقترح أسباباً (وزن منخفض، عمر صغير، جودة علف...).
- **تسعير المنتجات المصنعة**: نفس فكرة `suggest-product-price` المستخدمة في الكاترينج.
- **توقع الطلب**: من بيانات المبيعات التاريخية → كم دفعة تصنيع نحتاج الأسبوع القادم.

### المرحلة 6: التقارير

- ربحية كل دفعة ذبح
- التصافي اليومي/الشهري + مقارنة بالمعياري
- تكلفة وهامش كل منتج مصنع
- مخزون اللحوم الطازجة والمصنعة لحظياً
- تتبع كامل من رقم باركود إلى دفعة الذبح وحتى الطائر الأصلي

### المرحلة 7: التكامل مع الأنظمة الموجودة

- ربط بنظام **المنتجات** الحالي عبر الباركود الرسمي (الباركودات الـ 6224... التي رفعتها).
- ربط بنظام **الطلبات** → عند بيع منتج، يُخصم من المخزون الناتج عن دفعة معينة (FIFO).
- ربط بنظام **التنبيهات** → عند انخفاض مخزون لحم أو اقتراب صلاحية.
